1. Introduction

1.1 Purpose of Product

A one-paragraph description of your product's purpose

This product is developed for entertaining users through a fun series of puzzles and musical expression. The game will include a city that the user is able to explore and interact with. One unique aspect of this game is that the user is able to use an in-game recording studio to create their own soundtrack.

1.2

1.2 Scope of Product

This product will provide basic puzzles such as moving boxes to certain positions in order to advance in the game or gain access to certain areas or objects. The game will also give the player the ability to create their own OST for the game itself; the player will not always be able to play any note, however, in order to keep the OST from being repetitive and disharmonic.

1.3 Acronyms, Abbreviations, Definitions

OST - Original Soundtrack.

1.4 References

None

2. General Description of Product

2.1 Context of Product

This software will be a *.exe file that can be played on either laptop or PC. As a computer game, this product is intended to be played for entertainment purposes by users aged from 5 and older.

2.2 Domain Model with Description

User uses the Keyboard input to control the main character sprite, who will then interact with different objects; these objects could be pushable/pullable boxes, sheet music, or instruments. Most of these objects will only be accessible once the user manipulates different objects in certain ways; for example, moving the boxes will allow the user to pick up the sheet music objects, which in turn will unlock the ability to manipulate the OST via instruments.

2.3 Product Functions (general)

The product will function as a top-down, 2D puzzle game that incorporates musical elements into gameplay by utilizing multiple inputs and user interfaces.The game will incorporate multiple levels. Each level will reward one with another piece of sheet music. Once a sheet is collected, the player will be able to perform the piece and add it to the already-playing score.

The specific functions of the game include:

- Multiple interface inputs including accessibility features

- Evolving soundtrack

- Dynamic levels

- Saveable game files

2.4 User Characteristics and Expectations

The users that will play this game consist of all people age 5 and above. Different users will have a wide range of abilities depending on their age, background, and experience playing video games. Some users will be more experienced gamers while others will be complete newcomers to gaming. Text is an important part of our game so players will have to be able to read to play the game.

2.5 Constraints

We are a limited group with limited experience in game development. For the majority of our group this is our first time working on Unity so we must learn how to use this software while working on it.

2.6 Assumptions and Dependencies

None

4. System and Non-functional Requirements

4.1 External Interface Requirements

NF.4.1.1

PC - this game is only playable on PC.

NF.4.1.2

Mouse - the mouse is used in some puzzles in this game.

NF.4.1.3

Keyboard - the keyboard is used to move and play instruments.

NF.4.1.4

OS - the game is playable on both macOS and windows.

4.2 Performance Requirements

NF.4.2.1

Acceptable frame rate - Lagging would reduce the smoothness and appeal to the game.

NF.4.2.2

Predictable Mechanics - Continuity in mechanics that lack bugs would serve the game well.

4.3 Design Constraints

NF.4.3.1

Resources - Lack of computer hardware or developers could cause a slow or halt in development progress.

NF.4.3.2

Time - Not all mechanics or levels may be included due to a constrained time frame.

NF.4.3.3

Knowledge - Developers must learn to use software while also developing the game. Mechanics may not be all included due to limited Unity knowledge.

4.4 Quality Requirements

Our users have expectations that the game will be functional. This means it cannot crash or if it does, it must at least be possible to recover the game data from before the crash. Users also expect the UI including the map, inventory and menus to display everything correctly. The final expectation from the user is that they will be entertained from our product. Our system is not life-critical because the failure of this system will not result in the death or injury of any individuals.

NF.4.4.1

Data recovery- game data must be recoverable in the event of a system crash.

NF.4.4.2

UI- the user interfaces in the game, including inventory, menus, ect should properly display items based on gameplay.

NF.4.4.3

Entertainment value- the main reason for users to play our game is to be entertained, so the game must be interesting, exciting, and challenging.
